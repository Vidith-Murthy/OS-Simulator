A deep dive into one of OS's most notorious problems — when processes freeze forever, waiting for each other.

01

What is a Deadlock?

A deadlock occurs when a set of processes are permanently blocked, each waiting for a resource held by another process in the same set.

🔒

Real-World Analogy

Two cars on a narrow one-lane bridge, each waiting for the other to reverse. Neither moves forward, neither moves back. They are deadlocked.

💻

In Operating Systems

Process A holds Resource 1 and waits for Resource 2. Process B holds Resource 2 and waits for Resource 1. Both are stuck indefinitely — a classic deadlock.

⚠️

Why It Matters

Deadlocks can crash systems, cause data loss, or make applications completely unresponsive. Databases, OS kernels, and network systems must handle this carefully.

02

The 4 Coffman Conditions

All four must hold simultaneously for a deadlock to occur. Break any one, and deadlock is impossible.

1
🔐

Mutual Exclusion

At least one resource must be non-sharable. Only one process at a time can use it. If another requests it, that process must wait.

Printer: Only 1 process can use it at a time
2
🤲

Hold & Wait

A process holds at least one resource while waiting to acquire additional resources held by other processes.

P1 holds R1, waits for R2
3
🚫

No Preemption

Resources cannot be forcefully taken from a process. They are only released voluntarily after the process completes its task.

R1 cannot be taken from P1 forcefully
4
🔄

Circular Wait

A set {P0, P1, …, Pn} exists where P0 waits for P1, P1 waits for P2, …, Pn waits for P0.

P1→P2→P3→P1 (cycle)
03

Resource Allocation Graph (RAG)

A directed graph that visualizes which resources are held and requested by which processes.

P
Process Node (Circle) Represents an active process in the system
■
Resource Node (Rectangle) Dots inside = number of available instances
→
Request Edge (P → R) Process is requesting a resource it doesn't have yet
→
Allocation Edge (R → P) Resource has been allocated to this process

Deadlock Detection Rules

  • ✅No cycle in the graph → No deadlock possible
  • 🔴Cycle + single-instance resources → Deadlock exists
  • ⚠️Cycle + multi-instance resources → Deadlock may or may not exist — further analysis needed
04

Wait-For Graph & Detection Algorithm

The WFG simplifies the RAG by removing resource nodes, making cycle detection straightforward with DFS.

Derived from the RAG by collapsing resource nodes. If Pi requested Rk which is held by Pj, add edge Pi → Pj.

  • 1 Start with the Resource Allocation Graph
  • 2 Remove all resource nodes (rectangles)
  • 3 If Pi → Rk and Rk → Pj, draw Pi → Pj directly
  • 4 Run DFS — a back edge = cycle = Deadlock!
DFS Cycle Detection (used in the Simulator)
function dfs(node, visited, recStack):
    visited[node] = true
    recStack[node] = true

    for each neighbor of node:
        if not visited[neighbor]:
            if dfs(neighbor, visited, recStack):
                return true   // Cycle found!
        elif recStack[neighbor]:
            return true       // Back edge → Cycle!

    recStack[node] = false
    return false

🛡️ Deadlock Prevention

Prevent at least one of the four conditions from ever holding:

  • Eliminate Mutual Exclusion: Use sharable resources where possible (e.g., read-only files).
  • Eliminate Hold & Wait: Require processes to request all resources at once, or release all before requesting more.
  • Allow Preemption: If a process can't get what it needs, preempt its current resources and restart it.
  • Eliminate Circular Wait: Impose a total ordering on resources; processes must request in increasing order only.

🧭 Deadlock Avoidance

Requires advance knowledge of maximum resource needs. The OS keeps the system in a safe state — one where all processes can eventually finish.

  • Each process declares its maximum resource requirements upfront.
  • Before granting a request, OS checks if granting it keeps the system in a safe state.
  • If yes → grant. If no → make the process wait.
  • Common algorithm: Banker's Algorithm by Dijkstra.

🔍 Deadlock Detection & Recovery

Allow deadlocks to happen, then detect and recover:

  • Detection: Periodically run DFS cycle detection on the Wait-For Graph (WFG).
  • Recovery — Process Termination: Kill one or all deadlocked processes to break the cycle.
  • Recovery — Resource Preemption: Forcefully take resources from some processes and give to others.
  • The Simulate tab demonstrates this interactively!

🙈 Ignorance (Ostrich Algorithm)

Simply ignore the possibility of deadlock entirely.

  • If a deadlock occurs, the user restarts the system or application.
  • Used by most general-purpose OSes (Windows, Linux) in many practical situations.
  • Rationale: Deadlocks are infrequent enough that overhead of prevention/detection isn't always worth it.
  • Named after the myth of ostriches burying their heads in the sand.
